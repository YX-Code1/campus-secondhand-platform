# 校园二手交易系统 — 项目结构说明

**项目代号：** DB2026-RA-V1.0  
**技术栈：** Vue 3 + Spring Boot 3 + MyBatis-Plus + MySQL

本文档说明各目录、各包职责，以及主要类/文件的作用，便于阅读代码与答辩讲解。

---

## 一、整体目录

```
campus-secondhand/
├── backend/                 # 后端（Spring Boot）
├── frontend/                # 前端（Vue 3 + Vite）
├── sql/                     # 数据库脚本
├── docs/                    # 项目文档（本文档）
└── README.md                # 快速启动说明
```

**典型请求链路：**

```
浏览器 (Vue) → axios (/api/...) → Controller → Service → Mapper → MySQL
                                    ↑
                              Security + JWT 鉴权
```

## 二、后端 `backend/`

根包：`com.campus.trade`

### 2.1 根目录类

| 类 | 作用 |
|----|------|
| `CampusTradeApplication` | Spring Boot 启动入口，`main` 方法运行整个后端 |

### 2.2 包 `controller` — 控制层（接口入口）

接收 HTTP 请求、参数校验、调用 Service、返回统一 JSON。  
所有接口前缀为 `/api`（见 `application.yml` 的 `context-path`）。

| 类 | 作用 |
|----|------|
| `AuthController` | 认证：`POST /auth/register` 注册、`POST /auth/login` 登录 |
| `UserController` | 当前用户：`GET/PUT /users/me` 查询/修改个人资料 |
| `ItemController` | 物品：搜索、详情、热门、我的物品、发布/修改/删除、图片上传 |
| `TradeController` | 交易：发起购买、我的交易、按编号查询、更新交易状态 |
| `AdminController` | 管理端：统计、用户列表/禁用/删除、物品审核、强制下架 |
| `ExportController` | 导出：管理员 Excel 导出物品、交易记录 |

> **说明：** Controller 不写业务逻辑，只做「接请求 → 调 Service → 包装 `Result`」。

### 2.3 包 `service` — 业务层

核心业务逻辑；本项目 **未再拆 Service 接口 + Impl**，直接 `@Service` 实现类（Spring 常见写法）。

| 类 | 作用 |
|----|------|
| `AuthService` | 注册（查重、BCrypt 加密密码、手机号 AES 加密）、登录（Spring Security 认证 + 签发 JWT） |
| `UserService` | 个人资料查询/修改；管理员侧用户分页、禁用、删除 |
| `ItemService` | 物品 CRUD、搜索分页、热门榜、审核状态变更、敏感词校验、浏览量 +1 |
| `TradeService` | 创建交易单、状态流转（待处理/进行中/完成/取消）、买卖双方交易列表 |
| `AdminService` | 后台统计数据（交易量、用户数、待审核数、热门物品等） |
| `CacheService` | 缓存占位类：当前项目仅用 MySQL，方法为空实现，查询直接走数据库 |

### 2.4 包 `mapper` — 数据访问层（MyBatis-Plus）

**这里是接口**，由 MyBatis 在运行时生成实现，负责 SQL 与实体映射。

| 类 | 作用 |
|----|------|
| `UserMapper` | 表 `sys_user` 的增删改查 |
| `ItemMapper` | 表 `item` 的增删改查（含逻辑删除） |
| `TradeRecordMapper` | 表 `trade_record` 的增删改查 |

> 继承 `BaseMapper<Entity>` 即可获得常用 CRUD，复杂 SQL 可写在 `resources/mapper/*.xml`（本项目以注解/条件构造器为主）。

### 2.5 包 `entity` — 实体层（与数据库表对应）

| 类 | 对应表 | 作用 |
|----|--------|------|
| `User` | `sys_user` | 用户：账号、密码、姓名、手机、邮箱、角色、状态等 |
| `Item` | `item` | 物品：标题、分类、价格、描述、图片、卖家、审核/销售状态、逻辑删除 |
| `TradeRecord` | `trade_record` | 交易记录：交易号、买卖双方、金额、状态 |

### 2.6 包 `dto` — 数据传输对象

用于接口入参、出参，与 `entity` 分离，避免把数据库结构直接暴露给前端。

| 类 | 作用 |
|----|------|
| `LoginRequest` | 登录请求：用户名、密码 |
| `LoginResponse` | 登录响应：JWT token + 用户信息 |
| `RegisterRequest` | 注册请求：用户名、密码、姓名、手机、角色等（含校验注解） |
| `UserProfileRequest` | 修改个人资料请求 |
| `ItemRequest` | 发布/修改物品请求 |
| `UserVO` | 返回给前端的用户信息（不含密码） |
| `ItemVO` | 返回给前端的物品信息（含卖家昵称等） |
| `TradeVO` | 返回给前端的交易信息 |
| `StatsVO` | 管理端统计面板数据 |

### 2.7 包 `common` — 公共组件

| 类 | 作用 |
|----|------|
| `Result<T>` | 统一 API 响应：`code`、`message`、`data` |
| `PageResult<T>` | 分页结果：`list`、`total`、`page`、`size` |
| `BusinessException` | 业务异常（如「用户名已存在」），由全局处理器转为 400 |
| `GlobalExceptionHandler` | `@RestControllerAdvice`：捕获校验失败、认证失败、业务异常等并返回友好提示 |

### 2.8 包 `config` — 配置类

| 类 | 作用 |
|----|------|
| `SecurityConfig` | Spring Security：放行登录/注册/公开物品查询、JWT 过滤器、CORS、密码编码器 |
| `MybatisPlusConfig` | 扫描 Mapper、分页插件、自动填充 `createTime`/`updateTime`/`viewCount` |
| `WebConfig` | 静态资源映射：将 `/uploads/**` 映射到本地上传目录 |
| `DataInitializer` | 启动时若不存在 `admin` 用户则自动创建默认管理员 |

### 2.9 包 `security` — 安全与认证

| 类 | 作用 |
|----|------|
| `JwtAuthFilter` | 从请求头读取 `Authorization: Bearer <token>`，解析 JWT 并设置登录上下文 |
| `CustomUserDetailsService` | Spring Security 按用户名从数据库加载用户，封装为 `UserPrincipal` |
| `UserPrincipal` | 实现 `UserDetails`：供 Security 校验密码、角色、账号是否可用 |
| `SecurityUtils` | 工具类：获取当前登录用户、判断是否管理员 |

### 2.10 包 `util` — 工具类

| 类 | 作用 |
|----|------|
| `JwtUtil` | 生成/解析 JWT（用户 id、用户名、角色） |
| `CryptoUtil` | 手机号 AES 加密存储、解密展示 |
| `SensitiveWordFilter` | 从配置读取敏感词列表，发布物品时校验标题/描述 |

## 五、分层关系小结

| 层次 | 后端 | 前端 | 职责 |
|------|------|------|------|
| 表现层 | `controller` | `views` + `components` | 与用户交互 |
| 业务层 | `service` | `stores` + 页面内逻辑 | 业务规则 |
| 持久层 | `mapper` + `entity` | — | 读写数据库 |
| 横切 | `config`、`security`、`common`、`util` | `api/request.js` | 安全、异常、工具 |

**关于 Service 是否写接口：**  
本项目 Service 只有实现类，没有 `XxxService` + `XxxServiceImpl`。单一实现、结构简单时这样写完全合理；若将来有多种实现（如多种支付方式），再抽接口即可。
